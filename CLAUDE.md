# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Workflow

After every change you make in this repo, commit and push it (directly to `main` ‚Äî this is a personal single-author repo). No need to ask first.

## What this is

Personal macOS dotfiles: zsh config, utility scripts, and Claude Code config. No build system, no tests, no dependencies ‚Äî just shell files and a Bash install script.

## The symlink model (read this first)

The files **in this repo are the source of truth**. `install.sh` creates symlinks from `$HOME` back into the repo, so the home-directory copies and the repo copies are literally the same file:

| Repo file | Symlinked to |
| --- | --- |
| `.zshrc` | `~/.zshrc` |
| `bin/sleep-manager` | `~/bin/sleep-manager` |
| `claude/settings.json` | `~/.claude/settings.json` |

Consequences:
- Editing either side edits both. There is no "sync" or "copy back" step.
- `install.sh` is location-independent (it resolves its own dir via `BASH_SOURCE`), so the repo can live anywhere; re-run it after moving the repo to relink.
- Running `install.sh` backs up any real file in the way to `<name>.bak` before linking, and `rm`s a pre-existing symlink. Adding a new managed file means adding a `link` call in `install.sh`.
- **One deliberate exception to the symlink model:** `~/.zshrc.local` is a *real copy* (seeded by `install.sh` from `.zshrc.local.example`), not a symlink, so per-machine/private config (real `DEV_REPOS` paths, `TBEAM_HOST`, private completions) stays out of the repo. `.zshrc` sources it if present. Never commit a real `~/.zshrc.local`.

## Commands

```sh
./install.sh            # link all dotfiles into $HOME (idempotent; re-run to relink)
sleep-manager status    # show pmset timeouts, active sleep assertions, script-managed caffeinate
sleep-manager disable   # block sleep (sudo): pmset disablesleep 1 + backgrounded caffeinate
sleep-manager enable    # restore sleep (sudo): kill caffeinate + reset pmset defaults
```

There is no lint/test step. After editing a shell file, validate by sourcing it (`source ~/.zshrc`) or running the script.

## Component notes

- **`bin/csync`** ‚Äî `set -euo pipefail` Bash that two-way-merges Claude session history between `~/.claude/projects` and the iCloud Drive `claude-sessions` folder (`rsync -a --update`, no `--delete`, so machines converge to the union). Was a `.zshrc` function; extracted to a script (`help` still lists it ‚Äî it scans `~/bin`). `brctl download` is wrapped in `|| true` so its exit status never aborts the sync under `set -e`. **Periodic runs come from a `precmd` hook in `.zshrc` (`_csync_periodic`), not launchd** ‚Äî this is load-bearing and a dead end was hit getting there: iCloud Drive is TCC-protected and *background launchd agents are denied*, and on recent macOS (Tahoe) granting `/bin/bash` Full Disk Access has **no effect** (verified ‚Äî even a pure-bash `readdir` is denied under a launchd agent), because TCC won't pin an FDA grant to a platform interpreter running an arbitrary script. The shell hook sidesteps this entirely by running `csync` in the Terminal's already-approved context. The hook gates on a `~/.cache/csync-last-run` stamp (written *before* the run so overlapping shells don't double-fire), uses `$EPOCHSECONDS` (no `date` fork per prompt), and launches `csync` detached (`( csync ‚Ä¶ & )`) so it never blocks the prompt; background runs log to `~/Library/Logs/csync.log`. Don't reintroduce a launchd agent for this without a code-signed helper binary.
- **`bin/sleep-manager`** ‚Äî `set -euo pipefail` Bash with a `case` dispatcher. Sleep blocking is two layers: `pmset -a disablesleep 1` (OS-level, survives logout/reboot) plus a backgrounded `caffeinate -dimsu` whose PID is tracked in `/tmp/sleep-manager-caffeinate.pid`. Key gotcha documented in its `--help`: the caffeinate process dies on reboot but the pmset flag does **not** ‚Äî always run `enable` to fully restore; don't rely on a reboot. `status` is the source of truth for current state.
- **`.zshrc`** ‚Äî PATH (`~/bin` first), then `compinit` is run **before** sourcing anything that calls `compdef` (e.g. a completion sourced from `~/.zshrc.local`) ‚Äî order matters, this was a fix for a "command not found: compdef" error. Also defines `prview` (gh PR check summary via jq) and the interactive `nosleep` one-liner (the inline cousin of `sleep-manager`). **Per-machine config is split out:** `.zshrc` declares an empty `DEV_REPOS` and sources `~/.zshrc.local` (a real copy, not symlinked ‚Äî see the symlink-model note), which holds the real repo paths, `export TBEAM_HOST`, and any private completions. The committed `.zshrc` ships generic examples and derives all usage/error/completion strings from `${(k)DEV_REPOS}`, so it works (and stays generic) even with no local file. `install.sh` seeds `~/.zshrc.local` from `.zshrc.local.example`.
- **`tfind` (in `.zshrc`)** ‚Äî semantic search across saved Claude sessions: "which session was working on X?". Pipeline is **retrieve-then-rerank** ‚Äî a keyword pass over each transcript's title + your prompts builds a candidate pool (the same scan `_claude_sessions_fzf` uses; padded with recent sessions when keyword matches are thin, so divergent wording still reaches the model), then **Sonnet via `claude -p --model sonnet --output-format json`** reranks the candidates and returns a JSON `[{i, why}]` ranking. Two load-bearing details: (1) the model call is a **Python `subprocess.run(['claude', ‚Ä¶])`**, which execs the real `~/.local/bin/claude` binary and so **bypasses the zsh `claude` wrapper** (the wrapper's tpush sentinel would otherwise interfere); (2) **every failure path ‚Äî no CLI, timeout, unparseable reply, empty `[]` ‚Äî falls back to keyword order** so the fzf picker always populates and the command works offline. `tfind -k` forces the offline keyword path (no Sonnet). The keyword scorer (`hay.count(t) + 4*head.count(t)`) is recall-only here ‚Äî Sonnet does the precision ‚Äî so don't over-tune it.
- **`tbeam` (in `.zshrc`)** ‚Äî teleports a Claude session to another machine (default `$TBEAM_HOST`, set in `~/.zshrc.local`) and resumes it there; a **remote `tpush`**. Resolves a session exactly like `tpush` (current chat via `$CLAUDE_CODE_SESSION_ID`, else fzf-pick scoped to `$PWD`, `-a` for all), `rsync`s its `~/.claude/projects/<enc>/` dir over (`_tbeam_sync_transcript`, `--update`/no-`--delete` so the live source wins but a newer remote is never clobbered ‚Äî csync still reconciles in the background), then has the **host land it**. Landing logic (`_tbeam_land`) lives in the *shared* dotfiles so it runs on the far side via `ssh "$host" "TB_CWD=‚Ä¶ TB_SID=‚Ä¶ TB_MODE=‚Ä¶ zsh -lic _tbeam_land"` ‚Äî work passed in **`TB_*` env vars** (sidesteps nested-quote hell) and a **login+interactive** shell (`zsh -lic`) because Homebrew `tmux`/`fzf` are on the *login* PATH (`.zprofile`) while `claude` is on the *interactive* PATH ‚Äî plain `ssh host '‚Ä¶'` sees neither. Default mode reuses the host's own `_dev_resume_session` to make a first-class `dev-<repo>-<slot>` (so `tgo`/`tread`/`tpop` work there), then auto-attaches via `ssh -t`; `-f` resumes in the ssh foreground, `-d` leaves it detached with an attach hint. Two load-bearing constraints: (1) the repo must exist at the **same path** on the host (all repos live in `~/code` on every machine) ‚Äî `tbeam` guards with `ssh host test -d`; (2) from *inside* Claude it forces `-d` (a Bash subprocess has no TTY to `ssh -t` into). `_tbeam_land` is shared code, so after adding/changing it run `dots` on the host to pull it.
- **`tpush`/`tpop`/`claude()` wrapper (in `.zshrc`)** ‚Äî move a Claude session between a foreground terminal and a detached `dev-<repo>-<slot>` tmux session. **The load-bearing invariant: never two live `claude` processes on one session id.** Claude Code takes *no lock* on a session's `.jsonl` and does *not* auto-fork ‚Äî two live resumers of the same id both append with no coordination and the conversation diverges (the copy you aren't driving freezes). So: (1) `tpush` from *inside* Claude does **not** spawn `claude -r` itself ‚Äî it writes a `SPAWN\t<session>\t<cwd>\t<sid>` instruction to the `$CLAUDE_TPUSH_ATTACH` sentinel, and the `claude()` wrapper spawns the detached resume **after** the foreground exits (when the id is free) and then attaches. (`ATTACH\t<session>` is the already-backgrounded case; a plain-shell picker spawns inline since the picked session isn't live; an older shell with no wrapper spawns eagerly + warns.) (2) `tpop` captures the tmux session's claude PID (pane shell's child), `kill-session`s, then **waits for that PID to actually exit** (‚â§5s) before `claude -r` in the foreground ‚Äî `kill-session` only SIGHUPs, so resuming immediately raced the dying process. Was the "popped/backgrounded session stops updating while the original keeps going" bug.
- **`bin/pii-scan`** ‚Äî `set -euo pipefail` Bash + a Python heredoc (cashfwd's mining idiom) that fails if any PII appears in tracked/staged files. Reuses the **cashfwd two-layer ruleset**: (1) the **denylist** `scrub-rules.json` ‚Äî literal, case-sensitive personal identifiers, marked PRIVATE and **never committed here** (gitignored as defense-in-depth); read locally from `~/code/cashfwd-private/scrub-rules.json` (override `$PII_RULES`) and in CI from the `PII_SCRUB_RULES` secret; (2) the **ignore patterns** `pii-ignore-patterns.txt` ‚Äî regexes for documented false-positive *shapes* (no PII, tracked). Plus a dotfiles-only third layer: (3) the **allowlist** `pii-allowlist.txt` ‚Äî intentionally-public values for *this* repo (your GitHub handle; generic vendor names like `Anthropic`/`OpenAI` that the financial-repo denylist scrubs but are intrinsic to a Claude Code dotfiles repo). A denylist hit clears only if `value in allowEntry and allowEntry in line` (whitelisting a full email also clears its domain substring, but only on lines that contain the full email) or the value matches an ignore regex. **Don't edit the shared cashfwd denylist to silence a dotfiles false positive ‚Äî add to `pii-allowlist.txt` instead.** Two run modes: `--staged` (staged blobs via `git show :file`, used by the hook) and default (all `git ls-files`). **Fails open when the denylist is absent** (machines without cashfwd-private still commit) unless `--require-rules` (CI) is passed, so a missing secret is loud. Wired two ways: the tracked **`.githooks/pre-commit`** (enabled by `install.sh` via `git config core.hooksPath .githooks`) runs `--staged` locally; **`.github/workflows/pii-scan.yml`** runs `--require-rules` on push/PR to `main`, materializing the secret to a temp file. Bypass a local block once with `git commit --no-verify`.
- **`claude/settings.json`** ‚Äî symlinking this reproduces Claude Code setup on a new machine: `enabledPlugins`, `extraKnownMarketplaces`, and `permissions` are re-applied on first run (Claude re-clones marketplaces and reinstalls enabled plugins). MCP is *not* managed here: claude.ai connectors sync via the Anthropic account automatically, and local/stdio MCP servers live in the stateful, non-symlinked `~/.claude.json` (none today ‚Äî sync via a merge step if added, never commit `~/.claude.json`).
